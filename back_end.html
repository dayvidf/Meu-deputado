Contexto Técnico

A atuação de Backend Developer neste projeto foca na sustentação da infraestrutura de dados. O maior desafio superado foi a criação de um sistema Stateless que orquestra fontes de dados governamentais e motores de inteligência artificial de forma síncrona.

O foco não é apenas buscar o dado, mas garantir que ele chegue ao cidadão de forma rápida, filtrada e, principalmente, em linguagem acessível através do processamento de linguagem natural. Cada requisição do aplicativo dispara uma cadeia de processamento: autenticação com a API pública, filtragem bruta de campos desnecessários, agregação de categorias de despesas, e — quando há projetos de lei — extração e tradução automática do texto jurídico via LLM.

A escolha da arquitetura sem banco de dados foi deliberada: elimina custos de armazenamento, remove o risco de cache desatualizado e garante que o cidadão veja sempre o dado oficial mais recente do Portal da Transparência — alinhado com o princípio de transparência radical do produto.

Fluxo de Processamento
Pipeline de Dados · Passo a Passo

Como o Back-end Processa Cada Requisição

Sequência de 5 etapas executadas a cada chamada do aplicativo — sem estado persistido entre requisições.

ETAPA 01
Requisição
O aplicativo solicita informações de um deputado. O Back-end intercepta a chamada e identifica quais dados precisam ser buscados na API da Câmara.
ETAPA 02
Filtragem Bruta
Os dados chegam em formato bruto com centenas de campos desnecessários. O Back-end limpa tudo, retendo apenas nome, foto, partido e os valores financeiros relevantes.
ETAPA 03
Agregação
Gastos são agrupados por categoria (Combustíveis, Aluguel, Gabinete) e comparados com a média estadual para gerar o índice comparativo em tempo real.
ETAPA 04
Processamento IA
Se houver projeto de lei associado, o Back-end extrai a ementa, envia o texto jurídico para o modelo de linguagem e aguarda o resumo em linguagem cidadã.
ETAPA 05
Resposta Organizada
O Back-end reúne resumo da IA + dados financeiros + perfil em um único JSON leve e bem estruturado, entregando ao app o contrato final de dados.
Decisões de Arquitetura
Decisão Abordagem Adotada Justificativa Técnica Benefício
Persistência de Dados Arquitetura Stateless (sem DB) Sprint 1 foca em validação de produto; banco de dados adicionaria overhead operacional sem ganho imediato. ✓ Zero custo de storage; dado sempre atualizado
Fonte de Dados API REST da Câmara (Dados Abertos) Fonte oficial e pública; elimina custo de aquisição e garante dados auditáveis pela fonte primária. ✓ Confiabilidade e sem licenciamento
Processamento de IA LLM externo via API (OpenAI/Gemini) Evita custo de treinamento/hosting de modelo próprio. Modelos prontos têm performance superior em NLP jurídico. ✓ Time-to-market rápido; qualidade alta
Contrato de Dados JSON normalizado pelo Back-end Front-end não deve lidar com inconsistências da API externa. Back-end é o único responsável por entregar um contrato limpo. ✓ Desacoplamento e resiliência no Front
Segredos e Credenciais Variáveis de ambiente (.env) Chaves de API nunca expostas no código-fonte. Repositório público exige isolamento total de credenciais sensíveis. ✓ Segurança e conformidade

// ─── H.U. 01: Consulta de Perfil e Gastos ───────────────────────────────── FUNÇÃO buscar_perfil_e_gastos(deputado_id): // Tarefa 1 — Integração com a API da Câmara perfil_bruto = http.GET("api.camara.gov.br/deputados/{deputado_id}", headers={Accept: "application/json"}) despesas_brutas = http.GET("api.camara.gov.br/deputados/{deputado_id}/despesas", params={ano: ano_atual, pagina: 1}) // Tarefa 2 — Algoritmo de agregação por categoria categorias = ["COMBUSTÍVEIS", "ALUGUEL", "PASSAGENS AÉREAS", "GABINETE"] totais = {} PARA CADA despesa EM despesas_brutas: SE despesa.tipo ESTÁ EM categorias: totais[despesa.tipo] += despesa.valor // Tarefa 3 — Normalização e tratamento de nulos perfil = { nome: safe_get(perfil_bruto, "nomeCivil", "Nome não disponível"), foto: safe_get(perfil_bruto, "urlFoto", "placeholder.png"), partido: safe_get(perfil_bruto, "siglaPartido", "N/A"), estado: safe_get(perfil_bruto, "siglaUf", "N/A") } // Tarefa 4 — Lógica comparativa com média estadual media_estadual = calcular_media(perfil.estado) total_deputado = somar_valores(totais) delta_percentual = ((total_deputado - media_estadual) / media_estadual) * 100 RETORNAR { perfil, totais, delta_percentual } // ─── H.U. 02: Resumo de Projetos de Lei via IA ──────────────────────────── FUNÇÃO resumir_projeto_de_lei(proposicao_id): // Tarefa 1 — Pipeline de recuperação legislativa textos = http.GET("api.camara.gov.br/proposicoes/{proposicao_id}/textos") ementa = textos[-1].ementa // versão mais recente // Tarefa 3 — Prompt engineering estruturado system_prompt = """ Você é um tradutor de leis para o cidadão comum. Regras: use linguagem simples, máx. 3 frases, sem jargão jurídico, mantenha o sentido original. """ // Tarefa 2 — Orquestração do LLM via API resposta_ia = llm.POST( model="gpt-4o-mini", system=system_prompt, user="Resuma: " + ementa, max_tokens=150 ) // Tarefa 4 — Formatação do retorno em JSON limpo RETORNAR { titulo: extrair_titulo(ementa), resumo: resposta_ia.texto, palavras_chave: extrair_palavras(ementa) }
